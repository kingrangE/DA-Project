# 전사 회의실 예약 시스템

## 1. 전체 시스템 범위

전사 임직원이 회의실을 조회하고 예약·변경·취소하며, 관리자가 회의실과 운영 상태를 관리하는 시스템이다.

---
## 전사 회의실 예약 시나리오

1. 직원이 예약 시스템에 로그인한다.
2. 원하는 회의실을 검색한다.
    - 주소, 빌딩, 층, 회의실명
2. 회의실을 선택한다.
    - 장비, 수용인원 확인
3. 예약을 생성한다. (년, 월, 일 / 시, 분)
    - 예약 가능한 시간 확인
    - 예약 시작 시각 선택
    - 예약 종료 시각 선택
4. 참여자 작성
5. 요구사항 작성
6. 예약 확정
7. 예약 내역 확인
8. 예약 변경/취소

#### 요구사항
##### 조직 및 직원

1. 하나의 법인은 여러 부서를 가질 수 있다.
2. 하나의 부서는 반드시 하나의 법인에 속한다.
3. 하나의 부서는 여러 직원을 가질 수 있다.
4. 하나의 직원은 반드시 하나의 부서에 속한다.
5. 하나의 직원은 반드시 하나의 권한등급을 가진다.
6. 하나의 권한등급은 여러 직원에게 부여될 수 있다.
7. 하나의 직원은 반드시 하나의 재직상태를 가진다.
8. 재직상태에는 재직, 휴직, 퇴직이 있다.
9. 퇴직한 직원의 정보는 과거 예약 및 이력 보존을 위해 삭제하지 않는다.

##### 건물 및 회의실

10. 하나의 법인은 여러 건물을 가질 수 있다.
11. 하나의 건물은 반드시 하나의 법인에 속한다.
12. 하나의 건물은 여러 회의실을 가질 수 있다.
13. 하나의 회의실은 반드시 하나의 건물에 속한다.
14. 하나의 회의실은 반드시 하나의 회의실상태를 가진다.
15. 하나의 회의실은 반드시 하나의 최소권한등급을 가진다.
16. 하나의 권한등급은 여러 회의실의 최소권한등급으로 설정될 수 있다.



##### 계정

17. 하나의 직원은 여러 계정을 가질 수 있다.
18. 하나의 계정은 반드시 하나의 직원에게 속한다.
19. 하나의 계정은 반드시 하나의 계정유형을 가진다.
20. 하나의 계정은 하나의 역할만 수행한다.
21. 계정유형은 서로 배타적이며, 하나의 계정은 다른 계정유형의 기능을 수행할 수 없다.
22. 계정유형에는 다음 세 종류가 있다.
    * 일반사용자계정
    * 회의실관리자계정
    * 시스템관리자계정
23. 일반사용자계정만 예약을 생성·변경·취소할 수 있으며, 반드시 하나의 예약권한등급을 가진다.

24. 회의실관리자계정은 담당 회의실의 장비·비품 및 요구사항만 관리하며 예약 기능을 수행할 수 없다.

25. 시스템관리자계정은 전체 예약과 이력을 관리하지만 일반사용자로서 예약을 생성할 수 없다.


- 계정이 꼭 필요한 엔터티인가?
- 팀 토의 결과 하나의 직원이 하나의 계정만 가져야한다로 나왔다.
    - 현업에서 한 직원이 여러 계정을 갖는다는 것이 이상하고 보안상의 문제도 있기 때문이다.
- 그렇다면 계정과 직원이 1대1 관계인데 둘을 분리할 필요가 있나?
    - 없다. 따라서 직원에 계정이 흡수되어야한다.
- 계정이 직원에 흡수되면서 서브타입 역시 필요없어진다.
    - 한 직원이 여러 역할을 겸임할 수 있기 때문에 중복된 역할을 가지게 된다.
    - 역할 코드를 통해서 여러 역할을 바꿔가면서 수행한다.


##### 예약

23. 하나의 예약은 반드시 하나의 회의실을 대상으로 한다.
24. 하나의 회의실은 서로 다른 시간에 여러 예약을 가질 수 있다.
25. 하나의 예약은 반드시 한 명의 예약자를 가진다.
26. 하나의 직원은 여러 예약에서 예약자가 될 수 있다.
27. 하나의 예약은 반드시 하나의 계정을 통해 생성된다.
28. 하나의 계정은 0개 이상의 예약을 생성할 수 있다.
29. 회의 시작일시는 회의 종료일시보다 빨라야 한다.
30. 동일한 회의실은 시간이 겹치는 둘 이상의 활성 예약을 가질 수 없다.
31. 예약 변경 시에도 시간 중복 검사를 수행한다.
32. 예약은 활성 상태의 계정을 통해 생성한다.

33. 계정은 자신이 속한 직원을 식별한다.

34. 일반 예약의 경우 예약자의 예약권한등급이 회의실의 최소예약권한등급 이상이어야 한다.

35. 계정유형은 예약·회의실 관리·시스템 관리 기능에 대한 접근 권한을 결정한다.

36. 시스템 관리자가 다른 직원을 대신해 예약하더라도 기본적으로 예약자의 권한등급을 기준으로 검증한다.

##### 권한등급과 예약

32. 직원의 권한등급이 회의실의 최소권한등급 이상이어야 예약할 수 있다.
33. 예약 등록 또는 회의실 변경 시 권한등급을 검사한다.
34. 자신의 권한등급보다 높은 최소권한등급의 회의실로 예약을 변경할 수 없다.

##### 예약 참여자

35. 하나의 예약은 한 명 이상의 참여자를 가진다.
36. 예약자는 해당 예약의 참여자로 등록된다.
37. 참여자는 직원 중에서 지정한다.
38. 동일한 직원은 동일 예약에 중복 참여자로 등록될 수 없다.

##### 대리인

39. 하나의 예약은 대리인을 지정하지 않을 수 있다.
40. 하나의 예약은 여러 명의 대리인을 가질 수 있다.
41. 대리인은 직원 중에서 지정한다.
42. 예약자와 대리인은 서로 다른 직원이어야 한다.
43. 퇴직한 직원은 대리인으로 지정할 수 없다.
44. 대리인은 해당 예약을 변경하거나 취소할 수 있다.
45. 대리인이 회의실을 변경할 경우 대리인의 권한등급을 검사한다.


##### 예약 상태와 이력

46. 예약상태에는 최소한 예약됨과 취소됨이 있다.
47. 예약이 생성되면 현재 예약상태를 예약됨으로 설정한다.
48. 예약 생성 시 예약이력에 생성 내역을 기록한다.
49. 예약이 취소되면 예약의 현재 상태를 취소됨으로 변경한다.
50. 예약 취소 시 예약이력에 취소 내역을 추가한다.
51. 취소된 예약은 삭제하지 않는다.
52. 예약 변경 시 변경 내역을 예약이력에 기록한다.
53. 예약이력은 삭제하지 않는다.
54. 직원이 퇴직하면 해당 직원이 예약자인 미래 활성 예약을 취소한다.
55. 퇴직으로 취소된 예약도 예약이력에 취소 사유와 처리 내용을 기록한다.

## 4. 추후 검토 사항



- 재직 상태 이력을 따로 둘건지?
    방식 1: 이력에서만 현재 상태 관리
    - 직원에는 현재상태를 두지 않고, 종료일시가 NULL인 이력을 현재 상태로 본다.
    - 장점은 데이터 중복이 없다는 것. 단점은 현재 상태를 조회할 때 항상 이력 테이블을 확인해야 한다는 것.
    
    방식 2: 직원과 이력에서 함께 관리
    - 장점은 현재 상태 조회가 편하고 빠르다는 것. 단점은 두 값의 동기화를 보장해야 한다는 것.

    결론 : 예약 시스템에서 재직 상태이력을 관리할 필요가 없다.
    현실에서는 재직상태이력을 인사시스템에서 관리하지만 
    이번 과제에서는 재직상태이력까지 관리하는 이유가 없다.
        재직상태이력을 조회하는 경우
            1. 퇴직 또는 휴직 시 미래 예약을 자동 취소한다.    -> ?


- 참여자와 대리인 구분 명확히
    - 대리인 -> 대리자로 변경 요청
       - 예약자와 형식을 맞추기 위해.
- 예약 내역은 누가 가능한지?
    - 아무나? 예약자만? 참여자까지?


- 법인명 대신 별도의 법인ID를 주식별자로 사용할지 검토


- 현재 예약 사항과 겹치는 문제?
    - 현재 예약을 변경하고 싶은데 지금 예약과 겹쳐서 불가능한 문제
    - 12시~1시 예약을 변경한다 -> 12시~1시 예약을 취소한다 -> 12시반~1시반 예약을 생성한다.
        - 취소한 사이에 뺏긴다? 
        - 수동으로 취소하고 생성하게 하면 번거롭다.
            - 자동으로 취소되고 생성된다면?
                - 나의 예약과 겹칠 경우 예약이 가능해야한다.
                    - 예약을 생성할때 다른 예약과 겹치는지 확인을 언제하는지?
                - 내 예약은 12시~1시 예약이라서 12시~1시 예약은 꽉차있어서 예약이 불가능하다.


* 취소된 예약은 삭제하지 않고 상태를 변경하여 보존할것인지?
    - 상태를 update하는지 변경된 예약을 신규 저장하는지?
    - overwrite를 하는것이 아니라 신규 저장해야함.

## 해결된 검토 사항

- 회의실의 종류를 구분할 것인지?
    - 안한다
- 관리자 역할을 별도의 역할 엔터티로 분리할지 검토
    - 관리자 변경 이력을 관리하기 위해 분리

- 회의 시작·종료 시간을 날짜와 시간이 결합된 일시 형식으로 관리할지 검토
    * 예약 날짜 정보 단위 설정
        * 년월일 / 시분초 구분해서 저장
- 예약 이력 엔터티 추가 여부 검토
    이력은 로그로 저장한다.
* 언제 전까지 취소 가능하게 할건지?
    - 1초전, 회의중에도 취소 가능함
* 업무시간 외에 예약가능한지?
    * 24시간 가능한지?
        - 8시부터 20시까지
* 로그인 시스템?
    - 계정 엔터티를 새로 생성해서 관리


* 이력 데이터와 내역 데이터

이력은 기존의 속성이 변경되었다고 기록됨
내역은 과거에는 어땠는데 현재는 어떤지 변경되었다는 기록이 생성됨

즉 이력은 똑같은 인스턴스가 생길 수 있지만
내역은 언제 무엇이 어디서 어디로 변경되었다고 새롭게 생성된다.
