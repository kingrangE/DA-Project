# DA# 사용법

## 1단계 : 업무 규칙 정하기

    - 업무를 문장으로 정하기

    - 규칙에서 명사를 우선 추출

## 2단계 : DA#에서 새 모델
    - 논리 모델 : 어떤 데이터가 어떻게 관계되는지
    - 물리 모델 : 실제 구현할 테이블, 컬럼, 자료형

    - 초보때는 논리 모델 완성하고 물리 모델로 전환

## 3단계 : 주제영역 생성
    - 주제영역 : 관련 엔터티를 업무 단위로 묶는 장치
    예시) 정비예약관리, 고객관리, 차량관리 ...

    - 단순 화면 폴더가 아니라 업무 경계
    - 엔터티가 어느 업무 영역에 속하는지 논리적으로 설명

## 4단계 : 엔터티 생성
    - 논리명, 영문명, 정의
    - 업무객체가 아닌 시스템행위를 표현하는 것 경계

## 5단계 : 속성과 주식별자 정의
    - 고객번호는 '속성명', 고객을 유일하게 식별하는 '번호'는 속성 정의
    - 정의에서 이름을 반복해서 쓰지 않기

## 6단계 : 관계 연결
    - 관계 문장으로 변환
    - 선만 보지 말고 반드시 양방향 문장으로 읽어야함.

## 7단계 : 식별 관계와 비식별 관계
    - 비식별 관계 : 부모 식별자가 자식의 외래키
    - 식별 관계 : 식별자가 부모 식별자의 조합
    - 인조 식별자 : 사용 가능성

    - 선택 근거와 중복 방지 제약 설명

## 8단계 : 모델 검증
    - 구조 검증
        * 모든 엔터티에 주식별자가 있는가?
        * 엔터티 이름이 업무 객체를 표현하는가?
        * M:N 관계가 남아 있지 않은가?
        * 관계가 없는 고립 엔터티가 있는가?
        * 부모 없이 존재할 수 없는 자식의 관계가 적절한가?
    
    - 속성 검증
        * 한 속성에 여러 값이 들어가지 않는가?
        * 계산할 수 있는 값을 불필요하게 저장하지 않는가?
        * 코드와 코드명을 같은 엔터티에서 반복 관리하지 않는가?
        * 동일한 의미의 속성에 같은 명명 규칙이 적용됐는가?
        * 필수 속성과 선택 속성이 구분됐는가?
    
    - 관계 검증
        * 관계의 방향과 카디널리티가 업무 규칙과 일치하는가?
        * 선택 관계인지 필수 관계인지 설명할 수 있는가?
        * 식별 관계를 지나치게 사용하지 않았는가?
        * 부모 삭제 시 자식 데이터 처리 규칙이 있는가?



# 엔터티
## 분류 : key/main/action

### key
다른 엔터티의 존재 없이도 독립적으로 개체를 정의할 수 있는 엔터티.
판단 질문.

부모 엔터티가 아직 없어도 이 개체를 정의할 수 있는가?

예를 들면:

* 직원
* 부서
* 법인
* 건물
* 교과목

### Main 엔터티

부모 엔터티로부터 발생하지만, 해당 업무의 중심이 되고 여러 하위 엔터티를 가지는 엔터티.

판단 질문.

부모 없이는 정의되기 어렵지만, 생성된 뒤에는 하나의 핵심 업무 대상으로 관리되는가?

예를 들면:

* 계약
* 주문
* 카드
* 접수
* 단가

### Action 엔터티

실제 업무 행위나 사건이 발생한 결과를 저장하는 엔터티.
예를 들면:

* 카드 이용
* 수강 신청
* 주문 목록
* 계약 변경
* 공사 내역





## 표준

## 대상

해당 개체를 직접 표준화 대상으로 지정하는 값.

예를 들어 엔터티가 대상이면 그 엔터티는 상위 모델 설정과 무관하게 직접 표준화 대상으로 관리.

사용 예:

* 특정 엔터티만 반드시 표준화해야 하는 경우
* 상위 모델은 비대상이지만 일부 엔터티만 표준화하려는 경우
* 예외적으로 특정 테이블에 강제 표준화를 적용하는 경우

⸻

## 비대상

해당 개체를 직접 표준화 대상에서 제외하는 값.

사용 예:

* 임시 테이블
* 외부 연계 원본 테이블
* 로그성 적재 테이블
* 표준화하지 않기로 한 예외 엔터티

상위 모델이 대상이더라도 엔터티를 비대상으로 직접 지정하면 해당 엔터티는 표준화 대상에서 제외.

⸻

## 상속

상위 개체의 표준 대상 설정을 그대로 따르는 값.




## 발생주기
엔터티의 인스턴스가 추가되는 시점이 주기적인지 수시인지를
구분하여 선택
## 월간발생량 
매월 발생되는 엔터티의 평균 건수를 입력
## 보존기한(월)
 엔터티의 데이터를 관리해야 하는 기간(월)을 입력
## 총건수
보존기한 내에서 발생될 수 있는 엔터티의 최대건수를 입력하며,
월간 발생량과 보존 기한을 입력하면 자동으로 총 건수가 생성
(자동 생성된 건수는 직접 편집하여 수정해도 무방)


## 정의
엔터티가 의미하는 데이터의 집합적 성격 또는 생성 기준
(생성되는 시점 또는 주체) 등을 기술(정의된 내용에 따라 해당 엔터티의 독립성 여부를 판단하는 근거가 될 수 있습니다.)
## 데이터 처리 형태
 엔터티의 생성, 삭제, 수정 규칙 등을 입력
## 특이사항 
엔터티에 대한 특별한 내용이 있을 경우 입력
## 노트
엔터티 분석 과정에서 엔터티 별로 메모가 필요한 경우 내용을 입력



# 관계
## 관계선 편집

### 관계명
관계 명칭(3자 입장)을 입력
### 유형 
관계의 유형을 선택
### 타입
관계의 타입을 선택(1 : M , M : M, 1 : 1)

### 식별자 사용 
부모 엔터티 식별자의 상속 유형을 선택
### 비상속 처리 
부모 엔터티 식별자의 상속 여부를 선택
부모 엔터티로부터 상속 받은 PK(실질 식별자)를 더 이상 자식 엔터티에게 상속 시키지 않기
위해 해당 식별자(관계)를 비상속 처리하는 기능입니다.
① 비상속 시키고자 하는 식별자(관계)속성이 있는 관계를 더블 클릭합니다.
② 관계 다이어그램 윈도우에서 비상속 처리를 True 로 선택합니다.
③ 관계선이 비상속( )으로 표시되며 비상속으로 받은 엔터티는 자식 엔터티에 식별자가
비상속으로 처리되어 상속되지 않는 것을 확인할 수 있습니다.

## 부모 엔터티
관계동사 부모 엔터티 기준으로 자식 엔터티의 관계에 대한 정의를 기술
### 필수
부모 엔터티 기준으로 자식 엔터티의 선택사양 (Optionality)에 대하여 선택

## 자식 엔터티
관계동사 자식 엔터티 기준으로 부모 엔터티의 관계에 대한 정의를 기술
### 필수
자식 엔터티 기준으로 부모 엔터티의 선택사양(Optionality)에 대하여 선택

### 식별자 정보 
관계의 식별자 속성 리스트 정보를 확인
### 정의 
관계에 대한 정의를 기술
### 특이사항 
관계에 대한 특별한 내용이 있을 경우 입력


# 속성
## 속성명 
명명규칙을 준수하여 속성명을 입력
## 컬럼명 
명명규칙을 준수하여 컬럼명을 입력
## 실질식별자
최종 확정된 공식적인 식별자이며 본질식별자의 전부나 일부를 인조 속성으로 치환하여 실질식별자를 생성할 수 있습니다. DA#에서는 ‘#’으로 표시합니다.



## 본질식별자
실제 특정 개체가 생성되기 위해서 반드시 존재해야 하는 속성(들)을 말합니다. 
본질 식별자를 찾는 가장 확실한 방법은 ‘육하원칙’을 이용하는 것입니다. 
‘누가, 무엇을, 언제, 어디서, 어떻게, 왜’라는 이 여섯 가지의 질문을 통해 왜 이러한 행위가 발생했는지를 규명하면 정확한 사실을 나타낼 수 있습니다.
* 특별한 하자가 없다면 이것이 곧 실질 식별자
* 업무적으로 발생하며 해당 엔터티의 인스턴스를 유일하게 구분하는 데 참여하는가

## 핵심 
* 해당 속성이 없으면 주요 업무 처리·조회·판단이 어려운가
    생성일시·갱신일시·설명·사유는 보통 핵심 속성이 아님
    상태, 시작·종료 시점, 업무 대상 식별값은 대체로 핵심 속성
