# 요구사항 정의서

### 조직
1. 조직은 계층형 부서 구조를 가진다.
2. 직원은 하나의 부서에 소속된다.
3. 직원은 직급을 가진다.
### 자원 (회의실/비품)
1. 회사는 0개 이상의 건물을 가진다.
2. 회의실은 회의실명/건물명/층수/수용 인원/담당 관리자 정보를 갖는다.
2. 회의실은 특정 층에 속한다.
3. 회의실은 상태를 가진다. (사용가능/점검중/폐쇄)
4. 회의실은 여러 대의 비품을 갖는다.
5. 비품은 다른 회의실로 이동될 수 있다.
6. 회의실은 관리자가 관리한다. 
    - 관리자 : 회의실 전반적 상태 관리 및 장애 신고 용도
7. 장비는 예약에 개별 배정될 수 있다.


### 예약 
1. Back-to-Back 예약을 허용한다.
    - 끝나자마자 예약 허용
2. 반복 예약은 규칙으로 관리한다.
3. 반복 예약 중 중간 한 건만 개별 취소가 가능하다.
4. 반복에는 종료 일시를 설정해야 한다.
5. 반복 종료 일시를 변경할 수 있다.
6. 예약 시간은 표준 시간대로 저장하고, Application에서 변환한다.
7. 직원은 회의실을 예약하기 위해 자신에게 맞는 권한에서만 가능하다.
8. 예약 이력은 관리되어야 한다.
9. 두 사람이 같은 회의실, 겹치는 시간을 동시에 예약하면 나중에 예약한 사람이 거부되어야 한다.
10. 직원은 회사나 지역에 상관없이 예약이 가능하다.
11. 예약에 장비를 배정할 수 있다.
12. 예약에 참석자를 지정할 수 있다.
13. 예약 참석 여부는 따로 기록하지 않는다.
14. 예약

# 궁금한 점

- 회의실 예약 권한을 구분하려하는데, 이걸 사전(DB와 관계없음)으로 해야할지, 엔터티로 하나를 만들어서 해야할지 고민
    - 사전으로 하면 
        - 장점 : 유연하게 관리 가능 (ex, 1급 : A,B,C,d,..., 2급 ,f,e,o,i,... 이런식)
        - 단점 : 자동화 불가
            - 인스턴스 추가 시 자동 권한 부여 불가
    - 권한 엔터티로 관리하면
        - 장점 : 명확하게 관리 가능, 자동화 가능
        - 단점 : 유연하게 관리가 되지 않음, 인스턴스 추가 시 오버헤드 존재
- 반복 예약 규칙으로 관리 / 개별 건으로 application 관리
    - 규칙으로 관리하면
        - 장점 : 반복 규칙으로 개별 조회 가능
        - 단점 : 조회 성능 저하?
            - Exception이 많을 시 매번 그걸 다 조회해야 하나?
    - application에서 개별 생성을 여러 개 하는 방향으로 관리
        - 장점 : 조회 성능 향상 (일반 일정 확인과 같음)
        - 단점 : 규칙 따로 관리 어려움
