# 전사 회의실 예약 시스템 논리적 데이터 모델 설명

> 원본 모델: `DA_3조_산출물.model`  
> 작성 목적: DA 3조의 전사 회의실 예약 시스템 논리 모델의 구조, 설계 의도, 업무 규칙과 보완 사항을 설명한다.

---

## 1. 모델 개요

본 모델은 전사 임직원이 권한과 회의실 상태를 확인하여 회의실을 예약하고, 반복 예약·참여자·장비·상태 변경을 관리하는 시스템을 대상으로 한다.

논리 모델은 다음 사항을 중심으로 설계하였다.

- 법인과 소속, 직원의 조직 구조 관리
- 직원과 소속 조건에 따른 예약 권한 계산
- 건물과 회의실, 장비 및 회의실 상태 관리
- 단건 예약과 반복 예약 관리
- 예약자, 참여자 및 회의실 관리자 관리
- 직원 재직 상태, 회의실 상태, 코드 및 예약의 변경 이력 관리
- 공통 코드와 코드 유형을 통한 표준화된 상태값 관리

모델의 데이터는 크게 다음 두 범주로 구분한다.

1. **원천 데이터 허브 또는 MDM에서 관리되어 예약 시스템으로 제공되는 기준정보**
   - 법인
   - 소속
   - 직원
   - 건물
   - 코드 및 코드 유형
   - 권한 부여 규칙

2. **회의실 예약 시스템에서 직접 생성·관리하는 업무 데이터**
   - 회의실
   - 회의실 상태 이력
   - 장비 분류 및 회의실 장비 분류
   - 예약 및 예약 이력
   - 참여자
   - 반복 예약 규칙
   - 회의실 관리

## 3. 핵심 엔터티 설명

## 3.1 법인

기업집단 내 각 회사를 식별하는 기준정보 엔터티이다.

- 주요 식별자: 법인 코드
- 주요 역할:
  - 소속의 상위 조직 범위 제공
  - 건물 및 직원의 법인 귀속 기준 제공
  - 동일 기업집단 내 여러 회사의 데이터를 구분

법인은 소속보다 상위 개념이며, 하나의 법인은 여러 소속을 가질 수 있다.

---

## 3.2 소속

직원이 실제로 배치되는 본부·실·팀 등의 조직을 관리한다.

- 주요 식별자: 소속 코드
- 주요 속성:
  - 소속명
  - 법인 코드
  - 상위 소속 코드
  - 소속 단위 또는 소속 기능 관련 코드

소속은 상위 소속 코드를 이용하여 자기참조 계층을 구성한다.

```text
법인
 └─ 본부
     └─ 실
         └─ 팀
```

초기 설계에서는 `법인 코드 + 소속 코드`를 복합 식별자로 사용했으나, 코드 생성 정책을 변경하여 소속 코드 자체가 고유하게 생성되도록 설계 방향이 변경되었다. 따라서 최종 논리에서는 소속 코드를 단독 식별자로 사용하고 법인 코드는 일반 외래키로 두는 것이 일관된 구조이다.

### 순환 참조 방지

상위 소속 코드는 계층을 표현하지만, 속성을 추가하는 것만으로 순환이 자동 차단되지는 않는다.

- 자기 자신을 상위로 지정: CHECK 제약으로 차단
- `A → B → A`, `A → B → C → A`: 상위 소속 변경 시 재귀 조회를 통해 차단

---

## 3.3 직원

회의실을 조회하고 예약하는 사용자를 관리하는 기준정보 엔터티이다.

- 주요 식별자: 사번
- 주요 속성:
  - 직원명
  - 이메일
  - 내선번호
  - 소속 코드
  - 직위 및 관련 코드

직원은 하나의 소속에 귀속되며, 한 소속에는 여러 직원이 존재할 수 있다.

현재 소속 코드가 전사적으로 고유하다는 정책을 적용한다면, 직원은 소속 코드만 참조하고 법인은 `직원 → 소속 → 법인` 경로로 조회할 수 있다. 직원 엔터티에 법인 코드가 별도로 남아 있다면 초기 복합키 설계가 변경된 뒤 정리되지 않은 항목으로 볼 수 있다.

---

## 3.4 직원 재직 상태 이력

직원의 재직·휴직·퇴직 등 상태가 적용된 기간과 변경 사실을 관리한다.

- 주요 참조값: 사번
- 주요 관리 대상:
  - 재직 상태 코드
  - 상태 적용 시작 시점
  - 상태 적용 종료 시점
  - 변경 또는 생성 시점

재직 상태는 직원의 예약 자격과 직접 연결된다. 직원이 퇴직 상태로 변경되면 해당 직원의 미래 예약을 재검증하여 취소할 수 있다.

이 이력은 단순 과거 조회뿐 아니라 다음 근거로 활용된다.

- 퇴직 적용 시점 확인
- 퇴직자 미래 예약 자동 취소의 원인 추적
- 특정 예약 시점의 예약자 자격 검증
- 예약 상태 변경에 대한 감사 및 분쟁 대응

---

## 3.5 권한 부여 규칙

직원 또는 소속의 조건에 따라 예약 권한 등급을 산정하기 위한 규칙을 관리한다.

권한은 직원에게 단순 고정값으로 저장하기보다, 다음 기준에 따라 실시간 또는 조회 시 계산하는 구조를 지향한다.

- 직원 개인 조건
- 소속 단위
- 소속 기능
- 규칙 유효 기간
- 권한 등급 또는 권한 코드

유형별 대상 속성이 하나의 부모 엔터티에 모두 존재하면 조건부 NULL이 많아질 수 있으므로, 공통 속성은 권한 부여 규칙에 두고 대상별 속성은 서브타입으로 분리하는 것이 적절하다.

```text
권한 부여 규칙
 ├─ 직원 권한 부여 규칙
 ├─ 소속 단위 권한 부여 규칙
 └─ 소속 기능 권한 부여 규칙
```

## 3.6 건물

회의실이 위치하는 물리적 건물을 관리한다.

- 주요 식별자: 건물 코드
- 주요 속성:
  - 건물명
  - 주소
  - 법인 또는 관리 주체 정보

한 건물은 여러 회의실을 보유할 수 있다.

---

## 3.7 회의실

예약 대상이 되는 회의 공간의 기준정보를 관리한다.

- 주요 식별자: 회의실 코드
- 주요 속성:
  - 회의실명
  - 건물 코드
  - 층
  - 수용 인원
  - 요구 권한 등급 또는 관련 코드
  - 현재 상태

회의실 코드가 전사적으로 고유하다면 회의실 코드 단독으로 식별한다. 건물 내에서만 고유한 정책이라면 `건물 코드 + 회의실 코드`를 복합 식별자로 사용해야 한다. 모델 전체에서 이 식별 범위를 일관되게 적용해야 한다.

---

## 3.8 회의실 상태 이력

회의실의 사용 가능, 점검 중, 사용 불가 등의 상태가 적용되는 기간을 관리한다.

- 주요 참조값: 회의실 코드
- 주요 관리 대상:
  - 상태 코드
  - 상태 시작 일시
  - 상태 종료 일시
  - 변경 사유

회의실 상태는 현재 시점뿐 아니라 미래 시점에도 예정될 수 있다.

예를 들어 다음 주 월요일에 공사가 예정된 회의실은 현재 사용 가능하더라도 해당 시간대 예약 대상에서 제외되어야 한다. 따라서 현재 상태만 관리하지 않고 기간 이력을 보존한다.

활용 목적은 다음과 같다.

- 특정 예약 시간대의 회의실 사용 가능 여부 검증
- 상태 변경으로 인한 기존 예약 재검증
- 예약 자동 취소의 원인 근거 제공
- 과거 특정 시점의 회의실 운영 상태 확인

---

## 3.9 장비 분류 및 회의실 장비 분류

회의실에서 제공하는 장비 종류와 보유 관계를 관리한다.

### 장비 분류

- 대형 디스플레이
- 화상회의 장비
- 화이트보드
- 프로젝터

### 회의실 장비 분류

회의실과 장비 분류 사이의 다대다 관계를 해소하는 교차 엔터티이다.

- 권장 식별자: `회의실 코드 + 장비 분류 코드`
- 추가 가능한 속성:
  - 보유 수량
  - 사용 가능 여부
  - 설치 또는 반입 일자

예약자는 필요한 장비 조건을 입력하고, 시스템은 해당 장비를 보유한 회의실만 조회할 수 있다.

---

## 3.10 회의실 관리

특정 직원이 특정 회의실의 관리 업무를 담당하는 관계를 관리한다.

- 주요 참조값:
  - 사번
  - 회의실 코드
- 추가 가능한 속성:
  - 관리 시작 일시
  - 관리 종료 일시

한 직원이 여러 회의실을 관리할 수 있고, 하나의 회의실도 기간에 따라 여러 관리자를 가질 수 있다.

---

## 3.11 예약

직원이 특정 회의실을 특정 시간대에 사용하기 위해 생성한 핵심 업무 엔터티이다.

- 주요 식별자: 예약 ID
- 주요 속성:
  - 회의실 코드
  - 예약자 사번
  - 시작 시각
  - 종료 시각
  - 예약 상태 코드
  - 반복 예약 규칙 ID
  - 생성 및 갱신 시각

### 핵심 업무 규칙

1. 동일 회의실의 예약 시간은 중복될 수 없다.
2. 예약자는 예약 시점에 유효한 재직 상태여야 한다.
3. 예약자의 계산된 권한 등급은 회의실의 요구 권한 등급 이상이어야 한다.
4. 예약 시간대에 회의실이 사용 가능한 상태여야 한다.
5. 필요한 장비를 보유한 회의실만 예약 후보로 제공한다.
6. 퇴직 또는 회의실 사용 불가 등의 상태 변경이 발생하면 미래 예약을 재검증한다.

예약의 자연적 중복 방지 기준은 보통 `회의실 코드 + 시작 시각` 또는 시간 구간 중첩 검사로 구현한다. 종료 시각까지 자연키에 포함하는 것만으로 시간 중복을 완전히 방지할 수 없으므로 별도 구간 충돌 검증이 필요하다.

---

## 3.12 참여자

예약에 참석하는 직원을 관리하여 예약과 직원의 다대다 관계를 해소한다.

- 권장 식별자: `예약 ID + 사번`
- 주요 역할:
  - 예약별 참석자 목록 관리
  - 참석자 수 집계
  - 예약 변경·취소 통지 대상 제공

참여자에서 회의실 코드는 예약 ID를 통해 조회할 수 있으므로 일반적으로 중복 속성이다.

```text
참여자 → 예약 → 회의실
```

따라서 특별한 성능 또는 이력 보존 목적이 없다면 참여자에는 예약 ID와 사번만 두는 것이 적절하다.

---

## 3.13 반복 예약 규칙

여러 회차의 예약을 생성하기 위한 공통 규칙을 관리한다.

- 주요 식별자: 반복 예약 규칙 ID
- 주요 공통 속성:
  - 예약자 사번
  - 회의실 코드
  - 반복 유형
  - 적용 시작일
  - 적용 종료일 또는 반복 횟수
  - 예약 시작 시각
  - 예약 종료 시각

반복 규칙은 실제 예약과 분리한다.

- 반복 예약 규칙: 예약 생성 기준
- 예약: 규칙에 따라 생성된 실제 시간 슬롯

이 구조를 통해 반복 예약 중 일부 회차만 변경·취소하거나, 충돌한 회차만 제외하고 나머지를 생성할 수 있다.

### 반복 규칙 서브타입

#### 주 단위 요일 반복 예약 규칙

- 반복 예약 규칙 ID
- 요일 코드
- 반복 간격

#### 월 단위 일별 반복 예약 규칙

- 반복 예약 규칙 ID
- 월중 일

#### 월 단위 주별 요일 반복 예약 규칙

- 반복 예약 규칙 ID
- 월중 주차
- 요일 코드

부모 엔터티에는 공통 속성만 두고, 요일·월중 일·월중 주차와 같은 유형별 속성은 각 서브타입에 두어야 조건부 NULL과 중복을 줄일 수 있다.

## 3.14 예약 이력

예약의 생성, 변경, 취소 시점의 상태를 스냅샷 방식으로 저장한다.

- 주요 참조값: 예약 ID
- 주요 관리 대상:
  - 회의실 코드
  - 예약자
  - 시작 및 종료 시각
  - 예약 상태
  - 변경 시각
  - 변경 유형
  - 변경 사유
  - 변경 주체

예약 이력은 다음 목적을 가진다.

- 사용자 또는 시스템에 의한 예약 변경 추적
- 퇴직·회의실 상태 변경에 따른 자동 취소 근거 보존
- 특정 시점의 예약 상태 재현
- 장애·오류 발생 시 복구 근거 제공
- 사용자 문의 및 분쟁 대응

예약 이력에서 회의실 코드 등 원본 값을 반복 저장하는 것은 정규화를 위한 것이 아니라, 원본 예약이 이후 변경되더라도 당시 상태를 재현하기 위한 의도적 스냅샷이다.

## 3.15 코드 유형과 코드

업무 영역에서 사용하는 상태·구분·유형 값을 공통으로 관리한다.

### 코드 유형

코드 집합의 의미와 관리 단위를 정의한다.

예:

- 예약 상태
- 회의실 상태
- 재직 상태
- 반복 유형
- 권한 등급

### 코드

각 코드 유형에 속하는 실제 코드값을 관리한다.

예:

```text
예약 상태
- RESERVED: 예약
- CANCELED: 취소

회의실 상태
- AVAILABLE: 사용 가능
- INSPECTION: 점검 중
- UNAVAILABLE: 사용 불가
```

코드와 코드 유형을 분리함으로써 코드 유형명과 정의가 개별 코드마다 반복되는 것을 방지하고, 유형별 코드 체계를 독립적으로 관리할 수 있다.

## 3.16 코드 및 코드 유형 이력

코드 또는 코드 유형의 명칭, 정의, 사용 여부와 유효 기간이 변경된 사실을 보존한다.

활용 목적은 다음과 같다.

- 특정 시점의 코드 의미 확인
- 사용 종료된 코드가 포함된 과거 데이터 해석
- 코드명 또는 정의 변경 영향 분석
- 기준정보 변경에 대한 감사 추적

코드값 자체가 식별자로 사용되는 경우에는 코드값을 직접 변경하기보다 기존 코드를 사용 종료하고 신규 코드를 생성하는 방식이 안전하다.

## 5. 정규화 적용

본 모델은 비정규형 테이블을 먼저 작성한 뒤 단계별로 분해하기보다, 업무 개체와 함수 종속을 분석하여 처음부터 제3정규형에 가까운 구조를 도출하는 방식을 사용하였다.

### 5.1 제1정규형

다중값과 반복 그룹을 별도 엔터티로 분리하였다.

- 예약의 복수 참여자 → 참여자 엔터티
- 회의실의 복수 장비 → 회의실 장비 분류 엔터티
- 반복 규칙의 유형별 값 → 반복 예약 규칙 서브타입

### 5.2 제2정규형

복합 식별자의 일부에만 종속되는 속성이 없도록 엔터티를 분리한다.

예를 들어 코드와 코드 유형을 하나의 엔터티로 합치면 코드 유형명은 `코드 유형 번호 + 코드 번호` 전체가 아니라 코드 유형 번호에만 종속된다. 이를 코드 유형과 코드로 분리하여 부분 함수 종속을 제거한다.

### 5.3 제3정규형

일반 속성이 다른 일반 속성을 통해 결정되는 이행적 종속을 제거한다.

예:

```text
사번 → 소속 코드 → 소속명
```

직원에 소속명을 저장하지 않고 소속 코드만 참조함으로써 소속명을 소속 엔터티에서 일원화한다.

### 5.4 의도적 반정규화

현재값과 이력값을 동시에 저장하는 구조는 엄격한 정규화 관점에서는 중복이지만, 최신 상태 조회 성능을 위한 의도적 반정규화로 볼 수 있다.

- 직원의 현재 재직 상태 + 직원 재직 상태 이력
- 회의실의 현재 상태 + 회의실 상태 이력
- 코드의 현재값 + 코드 이력

이 경우 현재값과 이력의 현재 유효 행을 동일 트랜잭션으로 갱신하고 정합성 검증 기준을 명확히 해야 한다.

## 6. 이력 관리 원칙

본 시스템의 이력 엔터티는 공통적으로 과거의 상태와 변경 사실을 보존하여 이후 업무 판단, 감사 및 복구의 근거로 활용하기 위해 존재한다.

### 이력 관리 대상 선정 기준

- 여러 번 변경될 가능성이 있는가
- 변경된 과거값을 사후에 확인해야 하는가
- 변경이 예약의 생성·변경·취소에 직접 영향을 주는가
- 특정 시점의 유효성을 판단해야 하는가
- 장애·분쟁·감사 시 복구 또는 증빙이 필요한가

### 엔터티별 이력 목적

| 이력 엔터티 | 주요 목적 |
|---|---|
| 직원 재직 상태 이력 | 예약자의 자격 변화 및 미래 예약 취소의 원인 추적 |
| 회의실 상태 이력 | 예약 대상 시점의 회의실 사용 가능 여부 판단 |
| 코드 이력 | 시점별 코드명·정의·사용 상태 추적 |
| 코드 유형 이력 | 시점별 코드 분류 체계와 의미 추적 |
| 예약 이력 | 예약 변경 결과, 취소 사유, 감사 및 복구 |

### 저장 방식

이력은 변경 시점의 전체 상태를 저장하는 스냅샷 방식을 기본으로 한다.

스냅샷은 특정 시점의 상태 재현에는 유리하지만, 잘못된 입력을 자동으로 방지하지는 않는다. 잘못된 이력이 생성된 경우 기존 이력을 삭제·수정하기보다 정정 이력을 추가하여 원본과 정정 과정을 모두 남기는 것이 적절하다.

### 애플리케이션 로그와의 차이

애플리케이션 로그는 장애 분석과 실행 흐름 확인에 유용하지만, 업무 데이터와 별도의 처리 흐름으로 관리될 수 있다. 원본 변경과 이력 생성을 동일 데이터베이스 트랜잭션으로 처리하면 원본만 변경되거나 이력만 생성되는 불일치를 줄일 수 있다.

- DB 이력: 공식 업무 상태와 변경 근거
- 애플리케이션 로그: 실행 과정 및 장애 분석 보조 자료

### 보존 정책

이력은 지속적으로 증가하므로 사내 데이터 보존 정책을 적용한다.

- 최근 이력: 운영 DB 유지
- 장기 이력: 별도 이력 저장소 또는 아카이브 영역으로 이관
- 감사·분쟁 대응 데이터: 정책상 보존 기간 동안 유지
- 이관 이후에도 조회 가능하도록 메타데이터와 접근 경로 관리

## 7. 핵심 업무 시나리오

### 7.1 권한과 장비를 고려한 반복 예약

직원 A가 매주 월요일 오전 10시부터 11시까지 4주 동안 대형 디스플레이가 필요한 정기회의를 예약한다고 가정한다.

1. 직원의 사번과 소속을 조회한다.
2. 직원·소속 기능·소속 단위에 적용되는 권한 부여 규칙을 조회한다.
3. 유효 기간에 해당하는 규칙을 적용하여 직원의 권한 등급을 계산한다.
4. 직원 권한 이상으로 예약할 수 있는 회의실만 조회한다.
5. 회의실 장비 분류를 통해 대형 디스플레이 보유 여부를 확인한다.
6. 회의실 상태 이력을 통해 각 예약 대상 시간의 사용 가능 여부를 확인한다.
7. 기존 예약과 시간 구간이 겹치는지 검사한다.
8. 반복 예약 규칙을 생성한다.
9. 4개의 실제 예약을 각각 생성한다.
10. 충돌한 회차가 있으면 해당 회차만 실패 처리하고 나머지는 생성할 수 있다.

### 7.2 퇴직자의 미래 예약 자동 취소

1. 직원의 재직 상태가 재직에서 퇴직으로 변경된다.
2. 직원 재직 상태 이력에 퇴직 상태와 적용 시점을 기록한다.
3. 해당 직원이 예약자인 미래 예약을 조회한다.
4. 업무 정책에 따라 예약을 취소 또는 관리자 검토 대상으로 변경한다.
5. 예약 이력에 시스템 자동 처리 여부, 취소 사유, 처리 시점을 기록한다.
6. 재직 상태 이력과 예약 이력을 통해 상태 변경의 원인과 결과를 추적한다.

### 7.3 회의실 점검에 따른 예약 재검증

1. 회의실에 미래 점검 기간이 등록된다.
2. 회의실 상태 이력에 점검 상태와 적용 기간을 기록한다.
3. 해당 기간과 겹치는 미래 예약을 조회한다.
4. 예약을 취소하거나 대체 회의실 후보를 제공한다.
5. 예약 이력에 처리 결과와 회의실 상태 변경 사유를 기록한다.

## 8. 무결성 및 업무 제약

### 8.1 예약 시간 중복 방지

동일 회의실에 대해 다음 조건을 만족하는 예약이 존재하면 신규 예약을 제한한다.

```text
기존 시작 시각 < 신규 종료 시각
AND
기존 종료 시각 > 신규 시작 시각
```

단순 UNIQUE 제약만으로 시간 구간 중복을 완전히 방지하기 어렵기 때문에 DBMS의 범위 제약, 트리거 또는 트랜잭션 기반 검증이 필요하다.

### 8.2 상태 기간 중복 방지

한 직원 또는 한 회의실에 같은 기간 동안 두 개의 유효 상태가 존재하지 않도록 한다.

- 시작 시점 < 종료 시점
- 동일 대상의 유효기간 중복 금지
- 현재 상태는 하나만 존재

### 8.3 소속 계층 순환 방지

- 현재 소속과 상위 소속이 동일한 경우 차단
- 상위 계층을 재귀적으로 탐색하여 현재 소속이 발견되면 변경 거부

### 8.4 공통 코드 무결성

- 코드 유형에 존재하는 코드만 업무 엔터티에서 사용
- 사용 종료된 코드는 신규 데이터 입력에 사용하지 않음
- 기존 데이터의 해석을 위해 물리 삭제보다 사용 종료 처리

### 8.5 이력 정합성

- 원본 변경과 이력 생성은 동일 트랜잭션에서 처리
- 이력의 일반 사용자 UPDATE·DELETE 제한
- 오류 정정 시 기존 이력을 덮어쓰지 않고 정정 이력 추가

## 9. 모델에서 확인된 주요 보완 사항

다음 항목은 모델의 전체 방향을 변경하는 문제라기보다, 식별자 정책 변경과 모델 정리 과정에서 추가 확인이 필요한 사항이다.

### 9.1 직원의 법인 코드

소속 코드가 전사 고유하도록 정책이 변경되었다면 직원의 법인 코드는 소속을 통해 알 수 있으므로 제거 가능하다.

```text
직원 → 소속 → 법인
```

### 9.2 참여자의 회의실 코드

예약 ID가 회의실을 결정하므로 참여자에 회의실 코드를 중복 저장할 필요가 없다.

### 9.3 이력 엔터티의 유효 시작 시점

직원 재직 상태 이력과 회의실 상태 이력에는 상태가 언제부터 적용되었는지를 식별할 수 있는 시작 시점이 필요하다.

### 9.4 반복 예약 서브타입 식별자

각 반복 규칙 서브타입은 부모의 반복 예약 규칙 ID를 PK이자 FK로 상속하고, 필요한 경우 요일·주차·반복일을 추가 식별자로 구성해야 한다.

### 9.5 회의실 코드의 유일 범위

- 전사 유일: 회의실 코드 단독 PK
- 건물 내 유일: 건물 코드 + 회의실 코드 복합 PK

둘 중 하나로 업무 규칙과 모델을 통일해야 한다.

### 9.6 예약 이력의 추적 속성

완전한 감사 추적을 위해 다음 속성을 검토한다.

- 변경 유형
- 변경 사유
- 변경 주체
- 시스템 자동 처리 여부
- 원인 상태 이력 ID 또는 원인 발생 시점

### 9.7 DA.X 도구상 관계 메타데이터 확인

관계 삭제·재생성 과정에서 ERD 화면과 관계 상세 정보의 식별·비식별 설정이 일치하지 않을 수 있으므로 다음을 함께 확인해야 한다.

- 관계 상세 설정
- 자식 엔터티의 PK 전파 여부
- 실제 FK 속성 구성
- ERD 표기선

## 10. 설계 평가 요약

본 모델은 다음 점에서 논리적 모델링의 기본 원칙을 잘 반영한다.

- 조직, 자원, 예약, 공통 코드의 주제영역 분리
- 예약과 참여자, 회의실과 장비의 다대다 관계 해소
- 반복 규칙과 실제 예약의 분리
- 현재 상태와 상태 이력을 통한 시점별 업무 판단
- 공통 코드 구조를 통한 상태값 표준화
- 권한 규칙을 이용한 동적 예약 자격 계산
- 예약 이력을 통한 변경 추적과 복구 근거 제공

다만 식별자 정책이 설계 중 변경되면서 일부 기존 PK·FK 구조가 남아 있고, 이력의 적용 시점 및 감사 속성, 반복 규칙 서브타입의 식별 구조를 추가로 정리할 필요가 있다.

최종적으로 본 모델은 **제3정규형을 기본으로 하되, 최신 상태 조회와 감사 추적을 위해 일부 스냅샷 및 현재값을 의도적으로 보유하는 구조**라고 설명할 수 있다.
