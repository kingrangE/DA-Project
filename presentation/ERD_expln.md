# 발표 흐름

앞에서 설명드린 MDM 데이터가 실제 예약 과정에서 어떻게 사용되는지, 직원 A의 주간 정기회의 예약 시나리오를 통해 설명드리겠습니다.



## 4차 수정 - 최종본
먼저 구체적인 시나리오를 통해서 저희 시스템에서 어떻게 데이터가 흐르는지 보여드리겠습니다.
- 직원 A는 현대자동차 데이터아키텍처팀 소속이며, 팀 정기회의를 매주 월요일 오전 10시부터 11시까지 4주간 진행하려고 합니다. 이 회의에는 대형 디스플레이 장비가 필요합니다.

1. 시스템은 직원 A의 사번과 소속을 조회한다.
2. 사번·소속 기능·소속 단위에 적용되는 권한 부여 규칙들을 조회한다.
3. 해당 규칙을 적용하여 직원의 권한 등급을 실시간으로 계산한다.
4. 직원A의 권한으로 예약할 수 있는 회의실만 조회한다.
5. 회의실 상태 코드를 통해 사용 불가 상태인 회의실을 제외합니다.
6. 직원 A는 수용인원과 장비를 확인하면서 적절한 회의실을 선택합니다.
    - 자신의 권한등급으로 예약할 수 있고, 오전 10시부터 11시까지 예약할 수 있으며, 팀 전체 인원을 수용할 수 있고, 대형 디스플레이를 보유하고 있는 아반떼 회의실에 예약을 진행합니다.
7. 예약을 생성할때 먼저 반복 예약 규칙을 생성합니다.
8. 반복 예약 규칙은 여러 건의 예약을 업무 단위로 생성하고 관리할 수 있는 규칙입니다.
    - 직원 A는 4주동안 오전 10시부터 11시까지 월요일 아반떼 회의실을 예약하는 반복 예약 규칙을 입력합니다.
9. 4개의 실제 예약을 각각 생성한다.
10. 충돌한 회차가 있으면 해당 회차만 실패 처리하고 나머지는 생성할 수 있다.
11. 반복 예약 규칙으로 생성된 예약은 반복 예약 규칙ID가 추가로 입력됩니다.
12. 예약 일정을 변경한다면 반복 예약 규칙 ID에 의해 일괄 수정이 가능하다.
13. 변경된 내용은 예약 이력에 기록됨.

- 저희 팀은 지금 말씀드린 것과 같은 다양한 13가지 시나리오를 바탕으로 DB 성능 테스트까지 진행하여 모델링의 적절성을 검증하였습니다.
- 여기까지 고민과 토론을 멈추지 않은 3조 7WHY팀이었습니다. 감사합니다.


## 3차 수정
시나리오
직원 A는 현대자동차 데이터아키텍처팀 소속이며, 주간 팀 정기회의를 매주 월요일 오전 10시부터 11시까지 4주간 진행하려고 합니다. 이 회의에는 대형 디스플레이 장비가 필요합니다.

먼저 직원 A의 권한등급을 소속 기능 또는 단위에 적용되는 권한 부여 규칙에 의하여 실시간으로 계산합니다. 계산된 권한등급과 각 회의실의 요구 권한등급을 비교해, 직원 A가 예약할 수 있는 회의실만 조회 결과에 표시합니다.

추가로 회의실 상태 이력을 확인해 예약하려는 날짜와 시간에 사용 가능한 회의실만 조회합니다. 해당 시간에 사용불가 상태인 회의실은 제외합니다. 이후 직원 A는 수용 인원, 필요한 장비의 보유 여부를 확인하여 적절한 회의실을 선택합니다.
직원 A의 권한등급으로 예약할 수 있는 아반떼 회의실은 팀 전체 인원을 수용할 수 있고, 대형 디스플레이를 보유하고 있습니다.

그래서 직원 A는 4주동안 월요일 오전 10시부터 11시까지 아반떼 회의실에서 반복 예약을 생성하는 규칙을 입력합니다. 시스템은 이 규칙에 따라 4개의 예약을 생성합니다. 네 건 중 기존 예약과 충돌하는 일정이 있다면 충돌한 예약은 저장하지 않고, 충돌하지 않는 예약만 생성합니다. 이렇게 생성된 예약에는 반복 예약 규칙 ID가 추가로 입력됩니다.

이후 주간회의 시작 시간이 세번째주부터 갑자기 오전 11시로 변경되었다고 가정하겠습니다. 
그래서 직원 A는 세번째주 예약을 수정합니다. 이때 세번째주 예약은 반복 예약 규칙 ID가 존재하므로, 시스템이 반복 예약 규칙으로 생성되었다는 것을 알 수 있어 일괄 수정도 가능합니다. 변경된 2개의 예약 내용은 예약 이력에 추가로 기록됩니다.

저희 팀은 지금 말씀드린 것과 같은 다양한 13가지 시나리오를 바탕으로 DB 성능 테스트까지 진행하여 모델링의 적절성을 검증하였습니다.
여기까지 고민과 토론을 멈추지 않은 3조 7WHY팀이었습니다. 감사합니다.

### 시나리오 1차 수정
직원 A는 현대자동차 데이터아키텍처팀 소속이며, 주간 팀 정기회의를 매주 월요일 오전 10시부터 11시까지 4주간 진행하려고 합니다. 회의에는 대형 디스플레이 장비가 필요합니다.

직원 A가 회의실을 조회하면 시스템은 먼저 직원의 재직 상태를 확인합니다. 이후 소속 기능 또는 단위에 적용되는 권한 부여 규칙을 종합하여 직원 A의 권한등급을 실시간으로 계산합니다.
계산된 권한등급과 각 회의실의 요구 권한등급을 비교하여, 직원 A가 예약할 수 있는 회의실만 조회 결과에 표시합니다.

추가로 회의실 상태 이력을 확인하여 예약하려는 날짜와 시간에 사용 가능한 회의실만 조회합니다. 해당 시간에 점검 중이거나 사용불가 상태인 회의실은 제외합니다.
이후 직원 A는 수용 인원, 필요한 장비의 보유 여부를 확인하여 적절한 회의실을 선택합니다.
예를 들어 아반떼 회의실은 직원 A의 권한등급으로 예약할 수 있고, 예약 대상 날짜와 시간에 사용 가능한 상태이며, 대형 디스플레이를 보유하고 있습니다.

다음으로 직원 A는 4주동안 월요일 오전 10시부터 11시에 아반떼 회의실에서 반복 예약하는 규칙을 입력합니다. 시스템은 규칙에 따라 4개의 예약 인스턴스를 INSERT합니다. 이 과정에서 postgreSQL이 ON CONFLICT와 tstzrange타입과 exclude제약 조합으로 더블부킹을 막습니다. 이렇게 생성된 예약에는 반복 예약 규칙 ID가 추가로 입력됩니다.

이후 주간회의 시간이 세번째주부터 갑자기 오전 11시부터 12시까지로 변경되었다고 가정하겠습니다. 
사용자가 두 건의 예약을 각각 수정하는 대신 만들어둔 반복 예약 규칙의 예약 시작 시각과 예약 종료 시각을 수정합니다. 이 과정에서 예약 INSERT와 동일하게 postgreSQL제약에 걸리지 않는 예약들에 대해 일괄 변경합니다. 
변경된 각 예약의 변경 후 내용은 예약 이력에 기록합니다.



#### 수정 전
직원 A는 현대자동차 데이터아키텍처팀 소속이며, 직원 B와 C가 참여하는 주간 정기회의를 매주 월요일 오전 10시부터 11시까지 4주간 진행하려고 합니다. 회의에는 대형 디스플레이 장비가 필요합니다.

직원 A가 회의실을 조회하면 시스템은 먼저 직원의 재직 상태를 확인합니다. 이후 직원에게 저장된 고정된 권한등급을 사용하는 것이 아니라, 소속 기능과 소속 단위에 적용되는 권한 부여 규칙을 종합하여 직원 A의 권한등급을 실시간으로 계산합니다.
계산된 직원 A의 권한등급과 각 회의실의 요구 권한등급을 비교하여, 직원 A가 예약할 수 있는 회의실만 조회 결과에 표시합니다.

추가로 회의실 상태 이력을 확인하여 예약하려는 날짜와 시간에 사용 가능한 회의실만 조회합니다. 해당 시간에 점검 중이거나 사용불가 상태인 회의실은 제외합니다.

이후 직원 A는 수용 인원, 필요한 장비의 보유 여부를 확인하여 적절한 회의실을 선택합니다.
예를 들어 H03 건물 5층의 H03-5F001 회의실은 직원 A의 권한등급으로 예약할 수 있고, 예약 대상 날짜와 시간에 사용 가능한 상태이며, 참여자 3명을 수용할 수 있습니다. 대형 디스플레이를 보유하고 있습니다.

다음으로 직원 A는 4주동안 월요일에 반복 예약하는 규칙을 입력합니다. 반복 예약 규칙은 실제 일정 자체가 아니라 예약을 생성하는 기준입니다. 시스템은 앞으로 4주간 매주 월요일 오전 10시부터 11시까지 H03-5F001 회의실에 기존 예약과 충돌하는 일정이 있는지 확인합니다. 동시에 각 예약 대상 날짜와 시간에 회의실의 상태가 사용 가능한지도 확인합니다. 네 번의 일정이 모두 예약 가능하면 반복 예약 규칙을 생성합니다. 1주일마다 회의를 진행하므로 반복 간격은 1이며 월요일이라는 조건은 주 단위 요일 반복 규칙에 저장합니다. 
이 규칙을 바탕으로 4주간 회의할 네 개의 실제 예약이 생성되며, 각 예약에는 예약자 사번, 회의실, 시작 시각, 종료 시각과 예약 상태가 저장됩니다.

최종적으로 직원 A는 예약을 생성한 예약자이며, 직원 B와 C는 각 예약의 참여자로 등록됩니다.

이후 주간회의 시간이 오전 11시부터 12시까지로 변경되었다고 가정하겠습니다.
사용자가 네 건의 예약을 각각 수정하는 대신 만들어둔 반복 예약 규칙의 예약 시작 시각과 예약 종료 시각을 수정합니다. 시스템은 변경된 시간에 다른 예약과 충돌하지 않는지, 회의실이 사용 가능한 상태인지 확인한 뒤 수정 가능한 예약들을 일괄 변경합니다.
이때 과거 예약은 변경하지 않으며, 실제로 변경된 각 예약의 변경 전후 내용은 예약 이력에 기록합니다.
