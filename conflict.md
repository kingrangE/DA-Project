# 비품과 장비 구분
- 이관형 : 비품이 맞을까 장비가 맞을까?
    - 공공데이터 표준 사전과 Claude로 확인해보니 우리가 기존에 비품이라 정의했던 것은 장비에 더 맞다고 판단함.
    - 그래서 비품을 없애고 장비로 표준 사전을 정의하기로 함
- 전길원 : 장비로 변경하되, 비품은 비품대로 있어야 한다.
    - 멀티탭, 보드마카 이런 소모품을 어차피 회사에서 DB로 관리한다. 
- 전길원 : 그리고 회의실 예약할 때, 비품을 사용자들이 확인해서 직접 신청할 수 있게 하면 좋을 것 같다.
    - 이를 위해 비품 엔터티를 만들고, 예약 엔터티와 연결하는데, 비품 신청 내역 교차 엔터티로 M:M관계를 해소하자
- 이관형 : 예약 시스템에서 비품을 관리한다는 것은 현실과 맞지 않아질 가능성이 높고, DB에서 관리할 책임의 영역이 아닌 것 같다.
    - 만약 유저의 편의를 위한다면, 요청사항란을 만들어서 TEXT로 받는게 맞는 것 같다.
- 전길원 : 근데 비품을 회의실 예약할 때 안 보여줘도 될까?
    - 회의실과 비품을 연동할 필요가 있지 않을까?
- 이관형 : 회의에 꼭 필요한 요소들은 회의실에 있고, 장비 엔터티에서 관리한다. 그 외 소모품은 공동 창고같은데서 가져오는 걸로 정의함.
    - 이거를 DB에서 관리하는게 현실적이지도 않고 불편하다. 꼭 관리해야 할 데이터가 아니다.

    - 예약 엔터티에는 확실하게 현재 예약된 것만 관리해야 하는거 아닌가?

- 전길원 : 그렇지 않을까? 그렇게 해서 둘이 크기를 비교하면 되잖아.
- 이관형 : 근데 내가 찾아보니까 그게 아니라, 권한이 설정되는 규칙같은 것을 따로 만들어서 관리한대
    - 뭐 예를 들어, HR팀의 팀장급부터는 권한 2등급이다. 전사의 임원은 권한 1등급이다. 이런 식으로
- 전길원 : 음.. 그럼 이제 직위나 부서와 같은게 바뀔 때마다 그걸 검사해서 권한 속성값을 변경하는거야?
- 이관형 : 일단 직원 테이블에 권한은 속성값으로 가지지 않고, 매번 그냥 계산을 한대
    - 만약에 속성값으로 가지게 되면, 여러 권한에 포함되거나, 권한 규칙이 유효기간이 끝나서 바뀔 때마다 전직원에 대해서 다시 권한을 넣어야 하잖아.
        - 그건 굉장히 리소스가 크게 드는 작업이라서 그때그때 계산하는 방식으로 한대
- 전길원 : 아아 맞네
- 김용진 : 그러면 회의실 권한은 어떻게 하지 그것도 그떄그떄 계산해?
- 이관형 : 회의실 권한..? 회의실 권한이 왜 필요하지? 그냥 그 규칙의 결론에 포함되어야 하는거 아닌가
- 전길원 : 아니지 회의실도 원래 임원 회의실이었는데 일반 회의실이 된다거나 할 수 있잖아.
    - 그러니까 회의실도 뭔가 권한 값을 갖기는 해야지 변경될 수 있으니까
- 이관형 : 아 그럼 회의실은 고정된 권한 코드를 갖고, 유저는 그떄그때 계산하는 방식으로 하고 그러면 되겠다.

# 회의실이나 법인 등을 ID 인공키로 써야 하나?
- 전길원 : 아 근데 나는 ID를 인공키로 쓰는걸 최소화하고 싶어.
    - 일단 책에서 봤을 때 인공키를 쓰는건 하위 엔터티가 무거워지는걸 방지하기 위함 이런 느낌+익명화 느낌이었거든
    - 그래서 의미가 없는걸 최소화하고 싶은데
    - 그냥 이름으로 하면 안 되는 이유가 있을까?
- 김용진 : 그냥 이름으로 하면, 이름이 변경되었을 때 PK값을 다 바꿔야하는 문제가 발생한대, 그래서 변경될 수 있는 값을 PK로 하는건 별로 좋지 않대
- 전길원 : 아 맞네. 그럼 ID 대신 좀 인공적이지만, 사람이 읽을 수 있는 키면 좋을 것 같은데
    - 실제로도 우리가 사번 이런거 만들 떄도 K30017처럼 계열사 코드 + 무슨 코드 + 순서 번호로 이루어진단 말이야
- 이관형 : 그럼 실제로는 법인명, 소속, 사번, 건물, 회의실 이런게 다 그러한 코드로 관리가 이루어지는걸로 해야겠네
    - 근데 그 중에서 회의실을 빼고는 우리가 규칙을 관리할 필요가 없지 않나? 그러면 우리 DB에 적히면 안 되는거 아니야?
- 전길원 : 어 맞네 근데 회의실도 규칙을 관리할 필요가 없다고 봐. 회의실 정보도 예약 시스템에서 넣어주는게 아니잖아.
    - 회의실 정보는 예약시스템에서 넣는게 아니라 회의실이라는 자원을 관리하는 팀에서 정보를 등록하겠지.
    - 우리는 그 등록된 정보를 갖다 쓰는거고, 그러니까 MDM에만 그 코드 규칙 엔터티를 만들어놓고, 우리 테이블에는 기재하면 안 되는 것 같아

# VARCHAR v.s. TEXT
- 이관형 : 주소를 VARCHAR로 해야해?
    - postgre에서는 varchar랑 text가 같은 방식으로 저장되기 때문에 성능 차이가 없는데?
- 전길원 : 근데 TEXT로 하면 우리가 악의적인 입력에 대해서 대응이 안돼. VARCHAR로 하면 그 부분에 대해 제약이 들어가고
    - 만약 주소를 유저가 수기로 입력하는게 아니라 어떤 원천 시스템에서 받아오는거면 TEXT로 해도 괜찮다고 생각해
    - 그게 아니라면, VARCHAR로 스키마 구조 안정성을 지켜야 한다고 봐
- 김용진 : 일단 검색해본 결과 자신이 신뢰하지 못하는 데이터가 들어오는 곳에 TEXT를 쓰면 안 된다고 되어있긴 해
- 이관형 : 아 근데 VARCHAR 데이터타입으로 제약을 건다는게 뭔가 개념이 안 맞는다고 봐
    - 데이터 타입이라는건 우리가 스키마를 유지하기 위해 필요한건데 이걸로 제약을한다? 이게 안 맞는 것 같아.
- 전길원 : 그니까 너가 말한 그대로야. 우리가 스키마를 안정적으로 유지하려면 우리가 예상한 데이터가 엔터티에 기록되어야 하잖아.
    - 근데 TEXT로 하면 우리가 예상하지 못한 데이터가 입력될 수 있지.
    - 우리가 예상하지 못한 데이터가 악의적이든 휴먼 에러든 입력될 수 있는 구조를 만들어 놓은 것 자체가 데이터 무결성이 깨지는거란거지
- (좀 더 찾아보고 다시 얘기하기로 함)
- 이관형 : 음... 일단 행정안전부에서 VARCHAR를 사용하기는 하네. VARCHAR로 해도 될 것 같아
- 전길원 : 나도 찾아봤는데, PostgreSQL 공식 문서에서 VARCHAR(N)보다 TEXT+CHECK 제약을 권하네. 우리도 그렇게 하는게 좋을 것 같다.
    - 내가 생각하기에 우리가 VARCHAR/TEXT+CHECK를 선택하는 기준은 아래와 같이 가져가면 좋을 것 같아.
        - VARCHAR -> 개발자가 제약을 명확히 알고 있어야 하는 경우 (글자수 제한 등)
            - 스키마 단에서 구조를 알려주는게 필요한 경우
        - TEXT + CHECK -> 그 외의 문자열 입력

# 구조를 어떻게 하지
- 전길원 : 근데 이게 우리가 회의실 예약 시스템이지만, 결국은 사내 시스템이라 추가되는 사내 정보가 엄청 많단 말이야.
    - 그럼 이런 것도 다 표시해야 하지 않을까? 나는 각 부분들에 대해서 예약과 관련없어도 표시해야 하는 것 같은데
- 이관형 : 나는 우리 시스템에서 이용하지 않을 정보를 굳이 보여줘야 하나 싶어
    - 우리가 만드는 '예약'시스템 관점에서 생각해보면 퇴사 사유 이런건 필요없는거잖아. 그런건 없어야 한다고 봐
- 전길원 : 나도 이 시스템 관점에서는 없어도 되는데 결국 회사 내에 이런 정보는 있을거잖아. 
    - 우리가 기존 DB에 이걸 붙인다고 생각하면 난 있어야 한다고 봐
    - 우리가 만드는 이 예약 시스템 관련 엔터티들이 기존 DB에 들어가는건지 아니면 복제를 하는건지 이런걸 좀 알아봐야 할듯
- 이관형 : 우리가 만드는 시스템이 어떠한 기존 DB에 들어가는건 아닌 것 같아. 내가 생각하기엔 복제인 것 같아.
- 전길원 : 알아보니까 MDM Hub라고 하는 중앙 데이터 관리 시스템에 각 시스템에서 정보를 넣는대. 우리는 이제 그걸 가져와서 DB를 구성하는거지
    - 그래서 필요한 정보들을 일단위나 시간 단위로 동기화해가면서 쓰는거야
- 이관형 : 그럼 필요한 정보를 가져올 때 필요한 속성만 복사할 수 있으니까 "우리 DB에서는 우리 예약 시스템과 관련된 것만 보여준다."라고 설명해도 되겠네
    - 그럼 PK도 인공키를 가져올 필요가 없네
        - 레코드 단위로 우리가 필요한 정보만 가져오면, 그 인공키랑 순서가 안 맞을텐데 우리만의 인공키를 새로 만드는게 맞지
