1. 모델 범위와 시스템 경계
* “이 시스템이 직접 관리하는 데이터와 MDM에서 받아오는 데이터의 경계는 무엇인가요?”
    - 전사 공통으로 재사용되며 원천 시스템이 별도로 존재하는 법인, 소속, 직원, 건물, 공통 코드는 MDM(MDM, Master Data Management)에서 관리하고 예약 시스템에는 복제하여 사용합니다. 
    예약, 참여자, 반복 예약, 회의실 운영 상태 등 예약 업무 수행 과정에서 발생하는 데이터는 예약 시스템이 원천 데이터 주체가 됩니다.


2. 데이터 소유권과 원천 데이터 주체
* “원천 데이터가 변경되면 예약 시스템에는 어떻게 반영되나요?”
    - 기준정보는 원천 시스템을 기준으로 단방향 동기화합니다. 
    직원 퇴직이나 소속 변경처럼 예약 권한에 영향을 주는 정보는 지연을 최소화해야 하므로 이벤트 또는 CDC 방식이 적합하고, 일반적인 명칭 변경은 배치 동기화로 가능합니다.


3. 식별자 설계
* “사번을 직원의 PK로 사용한 이유는 무엇인가요?” 사번의 재사용 가능성, 법인 간 중복 가능성
    - 과제에서는 사번이 법인 내에서 유일하고 재사용되지 않는다는 업무 규칙을 전제로 자연 식별자로 사용했습니다. 
    그룹사 통합이나 사번 재사용 가능성이 있다면 법인 코드와 사번의 복합 식별자를 사용하거나 별도의 전사 직원 ID를 도입해야 합니다.


* “왜 예약에는 자연 식별자가 아니라 예약 ID를 사용했나요?”
    - 예약은 동일 예약자·회의실·시각 조합도 취소 후 재등록되거나 반복 예약에서 여러 건으로 생성될 수 있기 때문에 안정적인 자연 식별자를 구성하기 어렵습니다. 따라서 업무적으로 불변인 인공 식별자 예약 ID를 사용했습니다.


4. 조직 모델
* “소속, 소속 단위, 소속 기능은 각각 무엇이 다른가요?”
    - 소속은 현대자동차 인적자원관리팀처럼 실제 직원이 배치되는 조직 인스턴스입니다. 
    소속 단위는 본부·실·팀처럼 조직 계층의 유형을 의미하고, 소속 기능은 인사·생산·연구처럼 조직이 수행하는 업무 기능을 의미합니다.

* “소속 자기참조만으로 조직 개편 이력을 관리할 수 있나요?”
    - 현재 자기참조는 현행 조직 계층을 표현합니다. 조직 개편 시점별 구조를 재현하려면 소속 이력 또는 소속 계층 이력을 별도로 관리해야 합니다. 과제 범위에서는 현행 조직 구조만 포함했고, 이력은 확장 대상으로 두었습니다.


* “클로저 테이블을 쓰지 않고 자기참조를 선택한 이유는 무엇인가요?”
    - 조직의 직접 상하위 관계를 관리하는 것이 핵심이므로 우선 자기참조 구조를 선택했습니다. 
    전체 하위 조직을 빈번하게 조회하거나 계층 깊이가 크다면 재귀 쿼리 비용을 줄이기 위해 클로저 테이블을 추가할 수 있습니다.



5. 권한 모델
* “직원의 최종 권한 등급은 어떤 규칙으로 계산하나요?”
    - 소속 기능별, 소속 단위별 규칙 중 현재 유효한 규칙을 조회하고, 더 높은 권한 등급을 적용하는 방식으로 정의하였습니다.


* “권한 등급 숫자가 높을수록 권한이 높은가요?”
    - 과제에서는 숫자가 작을수록 높은 권한으로 가정했으며, 예약 가능 조건은 직원 계산 권한 등급이 회의실 요구 권한 등급 이상인 경우입니다.

* “권한을 실시간으로 계산하면 조회 성능 문제가 생기지 않나요?”
    - 직원의 소속이 변경될 경우 모든 직원 인스턴스의 권한이 수정되어야 합니다. 하지만 실시간으로 계산할 경우 그럴 필요가 없습니다.


* “직원이 소속을 이동하면 기존 예약 권한은 어떻게 처리하나요?”
    - 미래 예약에 대해 변경된 권한을 다시 검증해야 합니다. 
    권한이 낮아져 기존 예약 조건을 만족하지 못하면 자동 취소, 관리자 검토, 기존 예약 유지 중 업무 정책을 결정해야 합니다. 모델에는 권한 변경 시 미래 예약 재검증 규칙을 추가할 수 있습니다.


6. 예약과 반복 예약
* “반복 예약 규칙과 실제 예약은 어떤 관계인가요?”
    - 반복 예약 규칙은 반복 패턴을 정의하고, 실제 사용 일정은 개별 예약으로 생성합니다. 
    각 예약은 반복 예약 규칙 ID를 참조하므로 규칙에서 파생된 회차를 식별할 수 있습니다. 단일 회차만 취소하거나 변경해도 다른 회차에는 영향을 주지 않도록 설계했습니다.


* “반복 예약 중 한 회차만 변경하면 어떻게 표현하나요?”
    - 개별 예약은 반복 규칙과 별도로 수정할 수 있습니다. 
    변경된 회차는 기존 반복 규칙 ID를 유지하되 예외 여부 또는 원본 예약 시각을 추가해 반복 규칙과 다른 회차임을 구분할 수 있습니다.


* “동일 회의실의 시간 중복은 DB에서 어떻게 막나요?”
    - 논리 모델에서는 취소되지 않은 예약 간 시작·종료 구간이 중복되지 않는다는 업무 규칙으로 정의했습니다. 
    물리 모델에서는 일반 UNIQUE 제약만으로 기간 중복을 막기 어렵기 때문에 PostgreSQL의 exclusion 제약을 적용하였고 On conflict를 사용하여서 중단되지 않고 충돌하지 않은 예약은 성공하게 됩니다. 


8. 현재값과 이력값
* “회의실도 현재 상태와 상태 이력을 모두 저장하는 이유는 무엇인가요?”
    - 예약 가능 회의실 조회 성능을 위해 현재 상태를 회의실에 보관하고, 점검·폐쇄·운영 상태의 기간별 변경은 이력으로 관리합니다.


* "현재 상태와 상태 이력이 불일치하면 무엇을 기준으로 하나요?"



9. 코드와 코드 유형
* “예약 상태 코드와 회의실 상태 코드를 별도 엔터티로 만들지 않고 공통 코드로 통합한 이유는 무엇인가요?”
    - 코드 유형과 코드를 공통 구조로 관리해 코드 관리 방식을 표준화했습니다. 
    예약 상태와 회의실 상태는 서로 다른 코드 유형으로 구분하며, 업무 테이블에서는 코드 유형 번호와 코드의 조합을 참조합니다.


* “코드명이 바뀌면 과거 데이터 조회 시 현재 명칭이 보이나요, 당시 명칭이 보이나요?”
    - 현재 명칭을 보여주는 일반 조회는 코드 테이블을 참조하고, 특정 시점의 명칭이 필요한 감사·통계 조회는 코드 이력의 유효기간을 기준으로 조회합니다.


* “코드 유형 번호와 코드를 모두 FK로 전달해야 하나요?”
    - 코드 값이 코드 유형 내에서만 유일하다면 코드 유형 번호와 코드의 복합 FK가 필요합니다. 
    코드가 전사 전체에서 유일하다면 코드만 참조할 수 있지만, 코드 재사용성과 의미 충돌을 고려해 유형 번호와 코드의 조합을 사용했습니다.

12. 데이터 표준화 관점
* “ID, 코드, 번호를 어떤 기준으로 구분했나요?”
    - ID는 시스템 내부에서 개체를 식별하기 위한 인공 식별자, 코드는 업무적으로 의미를 가지며 외부와 공유 가능한 식별값, 번호는 업무에서 부여되고 순서 또는 식별 의미를 가지는 값으로 구분했습니다.



13. 물리 모델과 성능
* “가장 빈번한 대표 조회는 무엇인가요?”
    1. 직원의 미래 예약
    2. 특정 회의실의 일간 일정
    3. 특정 시간대 예약 가능 회의실
    4. 권한 등급별 예약 가능 회의실
    5. 직원의 반복 예약 규칙


* “예약 가능 회의실 조회는 조인이 너무 많지 않나요?”
    - 권한, 상태, 기존 예약, 장비 조건을 동시에 확인하기 때문에 조인 비용이 큽니다. 
    우선 정규화 모델로 정합성을 확보하고, 운영 단계에서는 현재 상태 컬럼, 계산 권한 캐시, 예약 가능 회의실 조회용 뷰 또는 파생 테이블을 검토할 수 있습니다.

14. 운영과 데이터 품질
* “데이터 품질 규칙은 무엇인가요?”
    - 예약 종료 일시 > 시작 일시
    - 유효 종료 일시 > 유효 시작 일시
    - 동일 회의실의 유효 예약 시간 중복 금지
    - 직원별 현재 재직 이력 최대 1건
    - 회의실별 현재 상태 이력 최대 1건
    - 현재 상태와 최신 이력 상태 일치
    - 반복 예약 적용 종료일 ≥ 시작일
    - 수용인원과 장비 수량은 0 이상
    - 상위 소속 자기참조 순환 금지
    - 코드 유효기간 중복 금지
